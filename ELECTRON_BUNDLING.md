# Bundling the App with Electron

This project ships as a desktop app by wrapping the Expo **web** build in
Electron. Electron also runs a WebSocket server so the UI can talk to the Unity
game. This guide covers running it in development and producing a distributable
installer.

```
repo root/
├── electron/
│   ├── main.js        # Electron entry: window + static server + WebSocket server
│   └── preload.js     # exposes window.unityBridge to the web app
├── frontend/          # the Expo app (its web export becomes the UI)
│   └── dist/          # generated by `npm run build:web` (git-ignore this)
├── game/              # put the Unity build here; bundled into the app
└── package.json       # root: Electron deps, build scripts, electron-builder config
```

---

## Prerequisites

- **Node.js 18+** and npm.
- Install root dependencies once (Electron, ws, electron-builder, etc.):

  ```bash
  npm install            # run in the repo root, NOT in frontend/
  ```

- The frontend has its own dependencies:

  ```bash
  cd frontend && npm install && cd ..
  ```

## Development (hot reload)

Runs the Expo web dev server and points an Electron window at it, so UI changes
hot-reload. The WebSocket bridge is live too.

```bash
npm run dev
```

What it does (see `scripts` in the root `package.json`):
1. `dev:expo` → `expo start --web` (serves on http://localhost:8081).
2. `dev:electron` → waits for that URL, then launches Electron with
   `EXPO_URL=http://localhost:8081` so `main.js` loads the dev server and opens
   DevTools.

If port 8081 is taken, change it in both the Expo command and `EXPO_URL`.

> On Windows, `npm run dev` uses `concurrently` + `wait-on`, which are installed
> as root devDependencies. If the inline `EXPO_URL=...` env syntax fails in your
> shell, run the two halves in separate terminals:
> ```
> cd frontend && npx expo start --web
> ```
> then in another terminal at the root:
> ```
> set EXPO_URL=http://localhost:8081   # PowerShell: $env:EXPO_URL="http://localhost:8081"
> npx electron .
> ```

## Production preview (no installer)

Build the static web export and launch Electron against it exactly as the
packaged app would (served over an internal localhost HTTP server):

```bash
npm run build:web      # expo export -> frontend/dist
npm run electron       # main.js serves frontend/dist and opens the window
```

Use this to confirm the production build works before making an installer.

## Building a distributable installer

```bash
npm run dist           # current OS: build:web + electron-builder
# or target a specific OS:
npm run dist:win       # Windows NSIS installer
npm run dist:mac       # macOS dmg
```

Output appears in `release/` (configured via `build.directories.output`).

### How packaging is wired

- `build.files` includes only `electron/**` — the Electron process code.
- `build.extraResources` copies:
  - `frontend/dist` → `resources/web`  (the UI the app serves)
  - `game`          → `resources/game` (the Unity build)
- At runtime `main.js` picks the right path automatically:
  - **packaged:** `process.resourcesPath/web`
  - **dev:** `frontend/dist`
- The UI is served over `http://127.0.0.1:<random-port>` instead of `file://`.
  This is deliberate: Expo's web export references assets with absolute paths
  (`/_expo/...`) that 404 under `file://`. The tiny static server in `main.js`
  resolves them and provides SPA fallback to `index.html`.

## The Unity game

- Drop your Unity standalone build into the `game/` folder. It gets bundled into
  `resources/game`.
- The game connects to the app's WebSocket server at
  `ws://localhost:8765?client=unity`. See
  [`game/UNITY_RECEIVE_FLASHCARDS.md`](game/UNITY_RECEIVE_FLASHCARDS.md) for the
  message format and a C# receiver.
- Sending flashcards: in the app open a Notebook → **Flashcards** tab → a deck
  with compiled cards shows a 🎮 button that sends that deck's JSON to the game.
  (The button only appears when running inside Electron.)

## Configuration reference

| Setting            | Where                         | Notes                                  |
|--------------------|-------------------------------|----------------------------------------|
| WebSocket port     | `electron/main.js` `WS_PORT`  | Default `8765`; keep in sync with Unity |
| App id / name      | root `package.json` `build`   | `appId`, `productName`                  |
| Installer targets  | root `package.json` `build`   | `win` / `mac` / `linux`                 |
| App icon           | root `package.json` `build`   | points at `frontend/assets/images/icon.png` |
| Dev web URL        | `EXPO_URL` env var            | only used in dev                        |

## Troubleshooting

- **White screen in production:** ensure `npm run build:web` ran and
  `frontend/dist/index.html` exists before `electron`/`dist`. The console
  (View → Toggle DevTools, or check terminal) shows the static-server path.
- **Backend calls fail:** the app still talks to your hosted backend
  (`BACKEND_URL` in `frontend/src/lib/api`). Electron doesn't change that.
- **Game can't connect:** the WebSocket server only runs while the Electron app
  is open. Start the app first, then the game (or add a reconnect loop in Unity).
- **Native modules:** none are required — `ws` is pure JS and the web SQLite
  shim is in-process, so there's no native rebuild step.
- **`localStorage` persistence:** on web/Electron the app stores its SQLite data
  and auth token in `localStorage`, which Electron persists per-app — so data and
  login survive restarts.
